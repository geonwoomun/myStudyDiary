1. 다음 각 항에 대하여 간단히 설명하라.
1) 경영: 일정한 목적을 달성하기 위하여 인적 물적 자원을 결합한 조직, 또는 그 활동

2) SWOT: Strength Weakness Opportunity Threat 강점 약점 기회 위협
경영 전략을 수립하기 위한 분석 도구이다. 내부 환경 분석과 외부 환경 분석이 있다.

3) 경영 활동에서의 핵심적인 요소: 경영방침, 경영이념, 경영목표, 경영 전략, 경영 전술

경영 이념을 최상위 단계로 결정하면, 이를 추구하기 위하여 '경영방침'이나 '경영지침'을 정하고, '경영방침'에 따라 달성해야 할 '경영목표'를 정한다. 또한 '경영목표'에 도달하기 위해 필요한 '경영전략'을 수립하고, '경영전략'을 실제로 수행하고 실천하기 위한 '경영전술'을 계획한다.

4) 프로젝트관리: 기술적인 부분들뿐만 아니라 프로젝트가 원활하게 진행되도록 프로젝트의 범위, 일정, 인력, 원가, 품질, 위험, 의사소통, 이해관계자, 조달 및 통합 등을 관리하는 부분으로 구성.

5) PM : Project Manager 프로젝트 관리자
6) 업무 메뉴얼: 기업에서 수행되는 모든 업무 기능들을 구체적이고 체계적으로, 그리고 명확하고 상세하게 규정하고, 업무를 수행하는데 필요한 각종 규칙이나 지침 등도 포함하여 작성하는 것이 좋다.

7) 업무 절차서: 기업에서 수행되는 모든 업무에 대하여, 수행되는 방식과 절차를 상세하고 구체적으로 필요하다면 예시를 사용하여 명세한다. 그리고 업무에 사용되는 문서들과 업무에 수반되는 데이터들도 포함하는 것이 좋으며, 신입사원이 업무 절차서만 이해하여 업무를 수행할 수 있는 수준까지 작성하는 것이 좋다.

8) 데이터 표준화: 데이터 표준화 미비로 인해 타 시스템의 데이터 활용이 곤란하여 자신에게 필요한 데이터를 추가적으로 정의함으로써 데이터 중복이 심화되고 데이터의 품질도 나빠진다. 이로 인해 정보 시스템의 풎리이 나빠져서 결과적으로 깅버의 생산성을 해치고, 기업 성장의 장애 요소로 작용.

9) RFP: Request For Proposal 제안 요청서 

10) CIO: Chief Information Officer 최고정보기술책임자

11) 정보기술위원회: 정보 시스템 관련 각종 표준들을 제정하고, 갱신하며 공포할 뿐 아니라 전 사원이 준수하도록 관리, 감독한다.

12) EAMS: 전사 아키텍처 관리시스템(Enterprise Architecture Managament System)

13) 전사: 기업이나 기관 자신이 가진 인력과 조직, 자원, 업무절차 및 기술 등을 총괄한 것.

14) 전사 아키텍처: 기업/기관을 전사적으로 관리하고 운영하기 위해 업무와 정보화를 체계적으로 추진하기 위한 종합 설계도 역할을 하는 구조

15) 추상화: 구체적이고 상세하게 표현되어 있는 개체들이나 행위들을 추상적으로 표현하는 과정으로서, 개념화라고도 한다.

16) 모델화와 모델: 업무에 사용되는 모든 실체들과 행위들 중에서, 정보 시스템으로 설계하고 구현하고자 하는 실체들과 행위들을 모두 모형화하고, 모형화한 것들의 상호 관련성과 상하 관계 등을 표현하는 과정을 모델화라고 하고, 모델화한 결과를 모델이라 한다.

17) 구조화와 구조: 모델화하는 과정에서, 개체와 행위들을 모형화하고 난 후, 모형들을 배치하고 서로 관련지어 주며 상화 관계 등을 구성시켜 주는 과정을 구조화라고 한다. 구조화한 결과를 구조라 한다.

18) 구체화: 추상화의 반대되는 개념으로서 추상적이거나 개념적인 내용을 실제 내용에 가깝게 상세하고 구체적으로 표현화는 과정이며, 상세화라고도 한다.

19) 전사 아키텍처의 관점 간 얼라인먼트: 전사 아키텍처의 동일한 영역에서 상위 단계의 관점과 하위 단계의 관점은 서로 추상화 수준이 다르더라도, 동일한 개념과 의미를 가져야 함을 의미한다.

20) 전사 아키텍처의 영역 간 얼라인먼트: 동일한 관점에서 서로 동일한 갠ㅁ이 적용되도록 해당 영역을 구성하는 것을 영역 간의 얼라인먼트라 한다.

2. 기업의 경영 시스템의 개념과 구조에 대하여 설명하라.
경영시스템은 경영 활동들을 개념적으로 구조화한 모습을 의미한다. 즉, 경영 활동을 구체적이고 상세한 모습으로 표현한 것이 아니라 개념적이고 추상적인 모습으로 표현한 것이다.

경영이념 - 경영방침 - 경영목표 - 경영전략 - 경영전술

3. 업무 조직을 종류별로 설명하고 장단점에 대하여 비교 설명하라.
기능 조직은 기업의 업무들을 성격이 다른 여러 기능들로 나누고, 업무 인력을 기능 단위로 배치하여 조직을 구성하는 방식이다.각 기능 조직에서 인력을 차출하여 프로젝트를 ㅅ수행하는 경우에는 기능 부서장보다 PM의 권한이 약하게 되어 부적당하다. 

프로젝트 조직은 프로젝트 단위로 조직을 구성하는 방식이다. 프로젝트가 형성될 때 해당 프로젝트 조직이 구성되고, 프로젝트가 끝나면 해체되어 다른 프로젝트 조직으로 개편되기 때문에, 프로젝트를 수행하는 중에는 pm의 권한이 막강하고 상호간에 의사소통이 원활하여 프로젝트를 수행하는데 적합하다. 하지만 프로젝트가 연속적이지 못할 때는 소속되었던 구성원들이 불안을 느끼고, 다른 프로젝트 팀에 소속될 때 다시 적응해야 하는 단점이 있으며, 특히 기업의 기능적인 업무를 수행하기에는 부적합한 구조이다.

위 두 가지 조직 구조를 혼합한 구조가 매트릭스 조직이다. 기능 업무를 강하게 하고 프로젝트 업무를 약하도록 구성할 수도 있고, 또는 이와 반대로 구성할 수도 있다. 기업에서 어떤 목적이 더 강한가에 따라 적절히 구성한다. 하지만 매트릭스 조직은 그 구성이 어려울 뿐 아니라 관리의 어려움이 많아 실제 사용되는 경우는 그렇게 많지 않다.


4. 정보 시스템의 현실에 대하여 설명하라
정보 시스템은 대부분 프로젝트로 구성되어 자체적으로 또는 외주에 의해 개발 되고, 인수된 정보 시스템은 해당 기업에서 자체적으로 또는 외주에 의해 유지보수된다. 그리고 사회가 급속도로 발전함에 따라 정보 시스템은 다양하고 복잡하게 되었지만, 고객의 요구를 충분히 충족시키지 못하고 있는 실정이다.

여러가지 원인으로 인해 의사소통이 원활하지 못하여 비효율적인 업무가 되거나 여러 부분에서 중복 투자하는 현상이 발생하곤 한다. 이와 같은 현상은 정보 시스템에 대한 유지보수 비용을 증가시키게 되고, 결국 경영의 비효율로 이어져 기업 발전을 저해하는 요소로 작용한다.

5. 정보 시스템에서 문서화 부족으로 인한 문제점과 해결책에 대하여 설명하라.
유지보수 작업이 어려워지기 때문에 많은 시간과 인력을 소요하여 결국 많은 비용을 지불하게 된다. 만약 정보 시스템에 대한 문서화가 전혀 이루어져 있지 않으면서, 일반적인 개발 방법론을 사용하지 않은 경우에는 초기 개발 비용보다 유지보수 비용이 더 많아지는 결과를 초래한다.

정보시스템이 이미 개발되어 유지보수 과정에서 문서화가 부족하다면, 별다른 해결책이 없다. 다진 능력을 가진 사원이 유지보수하고, 문서화하는 것이 최상책이다. 부족한 문서화에 대해 올바르게 다시 문서화를 한다는 것은 불가능할 뿐 아니라 엄청난 시간과 인력을 소요하여 올바르게 무선화하는 것도 어렵고, 문서화한 내용이 다시 사용될지 여부도 미지수이기 때문이다. 

정보 시스템을 효과적으로 개발하고, 유지보수 비용도 적게 소용되도록 하려면 문서화 계획에 따라 각종 문서들을 작성하고 협력 업체나 외주 업체에 관련된 업무도 기록하고 문서화하여야 한다.

6. 정보 시스템에서 표준화 부족으로 인한 문제점과 해결책에 대하여 설명하라.

표준화가 되어 있지 않다면, 동일한 업무라 하더라도 처리하는 사람에 따라 서로 다른 방법으로 처리하게 될 것이고, 동일한 의미의 단어나 용어를 서로 다르게 사용함으로써 의사소통에 문제가 발생하여 업무 처리에 혼선을 가져오게 된다. 즉 업무 표준화와 데이터 표준화가 부족하거나 표준화가 되어있다 하더라도 제대로 준수하지 않는다면, 비효율적인 업무 처리가 됨은 당연하다. 특히 데이터 표준화 미비로 인해 타 시스템의 데이터 활용이 곤란하여 자신에게 필요한 데이터를 추가적으로 정의함으로써 데이터 중복이 심화되고 데이터의 품질도 나빠진다. 이로 인해 정보시스템의 품질이 나빠져서 겨로가적으로 기업의 생산성을 해치지고, 기업 성장의 장애 요소로 작용하게 된다.

표준화로 인한 문제를 결하려면, 먼저 각종 표준화에 대한 방침이나 지침을 설정하고, 각종 표준화 계획을 수립한다. 표준화 계획에 따라 각종 표준화를 수행하고, 각종 표준화를 준수하면서 정보 시스템을 개발하고, 유지보수하는 것이 가장 효과적인 방법이다.

기업의 업무들도 표준화 되어야 일관성 있게 업무를 수행할 수 있다. 업무 절차서도 함께 작성하는 것이 좋다. 

정보 시스템을 효과적으로 개발하고 유지보수 하는 것이 목적이기 때문에 정보 시스템에서 사용하는 데이터들도 당연히 표준화되고 준수 되어야 한다.

7. 정보 시스템에서 부적절한 설계로 인한 문제점과 효과적인 정보 시스템의 개발에 대하여 설명하라.

인력 부족이나 시간 촉박 또는 비용 등의 이유로 인하여 이들을 제대로 준수하지 않고 개발하였다면, 많은 무제를 가지고 운영하기 때문에 유지보수 비용이 상당히 소요된다. 때로는 개발 비용 이상을 초과하는 경우도 종종 있다. 오류나 비정상 동작 등으로 인하여 정보 시스템이 올바르게 운영되지 않는다면, 업무가 제대로 수행되지 않는 것과 동일하기 때문에 비효율적인 업무 처리가 되는 것은 당연하다. 따라서 정보 시스템이 올바르게 운영되도록 정보 시스템의 운영 절차와 방법도 문서화되고 표준화되어야 한다.

8. 정보 시스템의 운영과 유지보수에 대하여 설명하라.


9. 전사 아키텍처의 관점에 대하여 설명하라.
전사 아키텍처의 추상화 수준을 의미하고, 추상화 수준이 높으면 높을수록 상위 단계의 관점이 된다.

10. 전사 아키텍처의 영역에 대하여 설명하라.
전사 아키텍처는 업무 아키텍처, 데이터 아키텍처, 애플리케이션 아키텍처, 기술 아키텍처로 나뉜다.

11. 전사 아키텍처 매트릭스에 대하여 설명하라.
전사 아키텍처의 각 아키텍처 영역을 네가지 관점에서 바라본 모습을 매트릭스 형태로 구성한 것이다. 즉 매트릭스 가로축을 전사 아키텍처의 각 영역으로 하고, 세로축을 전사 아키텍처의 각 관점으로 하여 형성된 각 셀에 해당 내용을 채워 넣는 형태이다.

12. 경영과 업무 및 전사 아키텍처 간의 연관 관계에 대하여 설명하라.
전사 아키텍처는 기업 입장에서 보면, 기억에서 수행되는 모든 업무 중의 하나에 불과하다. 정보 시스템으로 구현되지 않는 업무들도 기업에는 존재하기 때문이다. 즉, 전사 아키텍처는 기업 업무에 포함된다고 할 수 있다.
기업 업무는 경영이라는 좀 더 넓은 분야에 포함된다. 경영은 경제에 포함된다.

3) 핵심 줄기 업무
정보 시스템의 주된 목적을 달성하기 위해 수행되는 대상 업무

4) 업무친밀도
해당 업무들이 함계 수행되는 정도 또는 업무 의미의  유사성을 의미한다.

5) 업무 계층도
대상 업무에 대하여 계층 구조를 형성하여 도식화 한것.

6) 기능 차트
대상 업무들을 애플리케이션 관점에서 정보 시스템이 수행해야 할 기능들을 중심으로 업무를 표현한 것

7) 업무 명세서
PM이 먼저 발주 업체와 사용자인 고객의 원하는 내용을 문서화하는데, 이때 문서화된 내용을 업무 명세서라 한다.

8) 정보의 홍수
관련이 적은 내용들을 너무 많이 수집하는 것.

정보 시스템을 개발하기 위한 분석 방법론.
대표적으로 구조적 분석 방법론과 객체지향 분석방법론이 있다.

10) 관리대상 데이터
관리하는 데이터

데이터 타입에 데이터의 길이와 범위를 포함시킨 것을 말하는데, 데이터 타입은 숫자, 문자, 날짜 및 이진수 등과 같은 데이터의 종류를 말하고, 데이터 타입에 데이터의 길이, 소숫점 자릿수, 데이터의 범위 등까지 포함시키면 데이터 형식이 된다.

데이터의 값이나 모습을 지정한 규칙, 데이터 규칙에는 기본 값, 허용 범위, 데이터 포맷, 허용 값 등이 있다.

13) 데이터 요소

14) 엔터티

어떤 값을 대입하여 의미가 통함ㄴ 속서으로서의 가치가 있지만, 하나의 값만으로 의미가 통하지 않으면 엔터티일 가능성이 높다.

17) DB 환경 데이터
데이터베이스를 구축할 때 참조할 수 있는 환경에 관련된 데이터.

구축된 데이터베이스를 운영할 때 참조할 수 있는 정보 시스템 운영에 관련된 데이터로서, DB 환경 데이터, 백업과 복제 정보, 모니터링과 튜닝 정보, 권한 정보, 전사 데이터 관련 정보, 유지보수 정보, 관련 업체 정보, 사용자 정보 및 이해관계자 정보 등 수많은 정보들이 있다.

19) 데이터 품질
데이터에 대한 질, 품질을 의미한다. 데이터란 넓은 의미에서 보면, '기업에서 사용하는 모든 데이터', 즉 전사 아키텍처에서 사용되는 모든 데이터를 의미한다.

20) 데이터 표준화 원칙
데이터 사전과 데이터베이스 설계 표준을 구성하기 위한 기본적인 방침이나 원칙을 제시한 것.

사전적으로 문법상의 일정한 뜻을 가지는 말의 최소 단위를 의미하며, 데이터요소들의 명칭에 사용된 최소 의미 단위.

22) 표준 용어
표준 단어들의 조합으로 구성되는, 완전한 의미를 가지는 어휘를 말한다.

23) 코드 사전
데이터베이스와 애플리케이션에서 사용되는 코드들을 정의한 사전으로서, 엔터티를 기준으로 작성하는 것이 효율적이다.

24) 메시지 목록
애플리케이션에서 사용하는 메시지들을 모아 표준화하여 목록으로 구성한것.

25) 메시지의 종류
안내 메시지, 경고 메시지, 오류 메시지 등이 있을 수 있다.

2. 관리대상 업무에 대하여 성격과 구성에 대하여 설명하고, 종류별로 특징과 사례를 들어 설명하라.
모든 대상 업무는 트랜잭션 성격의 단일성 행위로 수행되어야한다. 여기서 트랜잭션 성격이라 함은 완전히 수행되든지, 전혀 수행됮 ㅣ않은 상태이든지, 둘 중 하나의 상태를 유지해야 함을 의미한다.

예를 들어 사원 채용은 대상 업무일까? 사원이라는 데이터도 관리해야하고 채용이라는 행위 자체도 관리할 필요가 있기 때문에 대상 업무라 할 수 잇따. 하지만 사원채용은 채용공고, 채용 접수, 채용평가, 채용확정등 여러개의 단일 행위들로 구성되어 있기 때문에 단일성 대상 업무가 아니라 다중성 대상 업무라 할 수 있다.

3. 관리대상 업무를 판단할 때, 주의해야 할 사항에 대하여 설명하라.

업무적인 측면만 고려하는 것이 좋다. 애플리케이션 측면이나 데이터 모델 측면은 고려하지 않는 것이 좋다. 애플리케이션 측면을 고려하게 되면 애플리케이션 기능이 고려되어 대상 업무가 잘못 도출될 수 있기 때문이다.

4. 업무 영역과 업무 주제 영역에 대하여 정의와 구성 방법에 대하여 설명하라.
업무 친밀도에 따라 대상 업무들을 그룹핑한 것을 업무영역이라 한다.
업무 영역은 계층적인 구성도 가능하다. 업무 영역을 계층적으로 표현할 때, 하위의 업무 영역들과 대상 업무들은 업무 친밀도가 높을수록 서로 가깝게 배치하고, 순서가 있는 경우는 순서를 고려하여 배치하도록 한다.

업무 영역들을 주제에 따라 분류하여 업무 주제 영역으로 그룹핑 할 수 있다. 물론 업무 주제 영역도 업무 영역과 마찬가지로 주제의 범위에 따라 좁게 또는 넓게 그룹핑될 수 있으므로 계층적 구조가 가능하다.

5. 다양한 요구사항 수집 방법에 대하여 설명하라
명세로부터 수집하는 방법: 업무 기술서, 업무 메뉴얼, 업무 절차서, 제안 요청서, 사업 수행 계획서 등과 같은 문서에서 수집한다.

기존의 시스템 개발에 사용되었던 문서들을 수집하는 방법 - 요구사항 분석서, 업무 기능 명세서, 프로세스 설계서, 데이터 흐름도, 화면 설계도 등과 같은 애플리케이션 개발에 사용되었던 문서들, 데이터 흐름도, 화면 설계도, 등등등의 문서들에서 수집

실무 현장에서 직접 수집하는 방법: 주문서, 송장, 전표, 판매 계획서, 조직도, 보안 매뉴얼 등의 장표들을 수집하거나, 현장에서 직접 체험하거나 면담 또는 설문 조사나 워크샵 등을 통하여 수집한다.

우수 경쟁사나 해외 기업들의 벤치마킹 책, 전문 잡지, 논문 등의 문헌, 세미나나 학회 등을 통해 수집하거나 해당 분야의 전문가나 경험자를 통해 수집할 수 있다.

6. 업무 명세서에 포함되어야 할 구체적인 내용에 대하여 설명하라.
업무 명세서에 포함되어야 할 내용에는 개발하고자 하는 시스템의 개요와ㅏ 목표, 핵심 업무 줄기에 해당하는 업무들, 핵심 줄기 업무를 보조하는 부수적인 업무들, 공통적으로 필요한 부수적인 업무들 등이 있는데, 이들을 명확하고 완전하게 서술하여야 한다. 또한 중요한 관리 대상 데이터들도 함께 서술하는 것이 좋다.

7. 요구사항을 분석하는 업무행위 기반 분석 방법론에 대하여 설명하라.
업무 명세서에서 동사를 추출하여 관리대상 행위를 중심으로 대상 업무들을 도출할 뿐 아니라 절차적으로 진행할 수 있기 때문에 초보자라 하더라도 비교적 용이하게 진핼할 수 있다. 더구나 데이터 모델링이 좀 더 효율적이 되도록 분석하기 때문에 데이터 모델링을 진행하는데도 도움이 된다.

8. 정보 시스템에서 사용되는 데이터의 명칭의 구성과 작명 방법에 대하여 설명하라.
데이터 명칭은 한긂ㅇ, 영문 약어명, 영문명 등으로 구성한다. 한글명은 주로 논리 데이터 모델에서, 영문 약어명은 물리 데이터 모델과 데이터베이스에서, 완전한 영문명은 참고용으로 작성한다. 데이터 명칭은 명칭 자체로서 자신의 의미를 파악할 수 있도록, 단순형의 업무 용어로서 데이터표준에 따라 작명하여야 한다.

너무 축약된 명칭은 지양하며, 업무나 사용자의 관점에 따라 의미가 달라질 수 있는 명칭은 수식어 등을 붙여 구체화함으로써 의미 전달이 충분하도록 작명해야한다.

9. 논리 데이터와 물리 데이터에 대하여 의미와 사례를 들어 비교 설명하라.
논리와 물리로 분류하는 것은 사용되는 곳과 표현 방법이 다르기 때문이다.
논리 데이터 타입은 데이터 종류로서 숫자, 가변길이 문자, 고정길이 문자, 날짜 등과 같이 표현하고, 물리 데이터 타입은 DBMS에서 사용되는 형식으로 표현한다. 그리고 논리 데이터 형식은 숫자10자리, 소수 2자리, 가변길이 문자 40자리, 고정길이 문자 1자리, 일자 등과 같이 표현하고, 물리 데이터 형식은 NUMBER(10, 2), VARCHAR2(40), CHAR, DATE 등으로 표현한다.

10. 식별자를 종류별로 의미와 특징에 대하여 설명하라.
식별자에는 본질 식별자, 실질 식별자 및 대체 식별자가 있다.
본질 식별자는 인스턴스를 생성할 때 인스턴스들을 본질적으로 식별하기 위한 속덩들로 구성되고, 실질 식별자와 대체 식별자는 생성된 인스턴스들을 식별하기 위한 속성들로 구성되는데, 실질 식별자는 실제 사용하는 식별자이며, 대체 식별자는 보조적으로 사용하는 식별자이다. 그리고 본질 식별자와 실질 식별자는 모든 엔터티에 필수적으로 하나씩 존재하지만, 대체 식별자는 엔터티에 따라 필요한 경우에만 하나 이상 선택적으로 존재한다.

11. 관계란 무엇이며, 업무 규칙의 4가지 요소에 대하여 설명하라.
관계란 정의된 기호를 사용하여 두 엔터티 간에 존재하는 업무적 관령성인 업무 규칙을 표현한 것으로, 데이터 집합과 데이터 집합들 사이의 관련성을 나타낸 것이기도 하다.

업무 규칙에 대한 4가지 규칙: 첫째 인스턴스들이 관계에 대하여 참여하는 정도를 나타내는 선택성 또는 참여도, 둘째 한 인스턴스에 대하여 상대 인스턴스가 몇 개나 관련되어 있는지를 나타내는 기수성, 셋째 한 인스턴스가 상대 인스턴스에게 어떻게 정보를 넘겨주는가를 나타내는 상속, 넷째 업무 관련성의 명칭을 나타내는 관계명이다.

12. 데이터베이스에서 사용되는 용어들에 대한 수학적인 표현, 파일 처리 표현, 논리 데이터 모델 표현 및 물리 데이터 모델 표현 등을 비교 설명하라.
데이터베이스를 구축하려면, 논리 데이터 모델을 구성하여 물리 데이터 모델로 변환하고, 데이터베이스로 구축해야 한다. 동일한 의미이지만, 발전 과정과 데이터베이스 구축 과정에서 서로 다른 용어로 표현하는 경우가 많다.

13. 단어 사전과 용어 사전이란 무엇이며, 각각 구성 방법에 대하여 설명하라.
표준 단어들을 정의한 사전으로서, 이음동의어와 동음이의어의 처리 방법, 단어의 영문 약어명을 정의하는 방법,, 접미어와 접두어의 사용 방법 등은 데이터 표준화 원칙에 정의되어 있어야 하고, 데이터 표준화 원칙에 정의되어 있는 내용을 기준으로 단어 사전을 구성하여야 한다.

용어 사전은 이러한 표준 용어들을 정의한 사전으로서, 영문 약어의 조합 방법, 데이터 요소들의 명칭 부여 방법 등을 데이터 표준화 원칙에 정의할 수 있고, 이러한 데이터 표준화 원칙을 기준으로 용어 사전을 구성하여야 한다.

테이블 행이 삭제되면 테이블에서는 해당 블록의 블록 헤어데 있는 슬롯 값을 null로 하여 데이터가 없음을 나타내고, 해당 인덱스들에서는 인덱스 엔트리에 삭제 표시자를 저장하여 인덱스 엔트리가 무효임을 표시한다.

31) 데이터 아키텍트와 DBA

DBA는 구축된 데이터베이스 전반에 대해 관리하는 사람이다.

32) 데이터 관리 프로세스

33) SQL
Structured Query Language은 애플리케이션이나 데이터베이스 관리자 또는 데이터베이스를 액세스하고자 하는 사용자가 DBMS에게 내리는 질의어 형태의 명령어이다.

34) 커서 방식의 처리
커서 방식의 처리에서는 처리하고자 하는 대상 집합을 먼저 커서로 선언한다. 그리고 필요한 경우에 해당 커서를 오픈하면, 대상 집합이 커서에 저장된다. 이 때 커서의 한 행씩 페치하여 사용한다. 사용이 끝난 커서는 닫아 줌으로써 커서 동작이 종료된다.

2. 데이터베이스 데이터의 특징에 대하여 설명하라.
정보 시스템에서 사용되는, 관리대상의 업무 데이터들로써 최소의 중복과 통제된 중복을 가진 통합된 데이터이고, 접근 가능한 디스크와 같은 저장 매체에 저장된 저장 데이터이다. 그리고 기업의 업무 기능들을 수행하기 위해 필요한 운영 데이터로서, 여러 명이 공동으로 소유하고 유지하며 사용하는 공용 데이터이다.

3. 데이터베이스의 특징에 대하여 설명하라
데이터베이스는 DBMS에게 SQL문을 사용하여 질의하면 실시간으로 처리하여 응답해주는 실시간 접근성,  실제 업무에서 사용되고 있는 데이터들을 실시간으로 삽입하고 갱신하며 삭제하는 동적인 지속적 변화, 여러 명이 함께 접근하여 사용할 수 있는 동시 공용, 위치나 주소가 아닌 데이터 값에 따라 참조하는 내용에 의한 참조 방식, 대용량 데이터들을 처리 등의 특징을 가지고 있다.

추상화 수준에 따라, 데이터 모델은 개괄 데이터모델, 개념 데이터 모델, 논리 데이터 모델, 물리데이터 모델 등으로 분류할 수 있다.

5. 데이터 모델의 피터 첸 표기법에 대하여 설명하라.
고전적인 표기법으로 교과서 등에서 많이 사용되고 있으나 실무에서는 거의 사용되지 않는다. 피터첸 표기법에서는 엔터티를 강한 엔터이와 약한 엔터티로 분류하고 있는데, 각각 단일 사각형과 이중 사각형으로 표현하고, 엔터티명은 사각형 내부에 표기한다.

6. 데이터 모델의 바커 표기법을 DA# 모델링 퉁에 준하여 설명하라
엔터티를 사각형으로 표현하고 엔터티명은 사각형 내부 또는 외부에 표기한다. 속성은 엔터티 내부에 나열하여 표현하는데, 필수 속성은 속성명 앞에 *을 선택 속성은 O를 실질 식별자는 #을 대체 식별자는 (#)으로 표기하고, 관계 속성은 관계 속성명 뒤에 (FK)을 표기한다. 그리고 관계는 두 엔터티 사이에서 선으로 연결하여 표현하는데, 필수 참여를 나타내는 필수 관계는 실선으로, 선택 참여를 나타내는 선택 관계는 점선으로 나타낸다. 1의 기수성은 어떤 표기도 하지 않고, M의 기수성은 까마귀 발을 표기한다. 식별 상속은 관계선에 bar을 표기하고, 비식별 상속은 어떠한 표기도 하지 않는다. 관계명은 관계선 위에 표기한다. 그리고 서브타입은 엔터티 내부의 사각형으로 표현하는데, 하나의 서브타입셋은 내부에 자신의 서브타입들을 묶어 사각형으로 표현한다. 물론 서브타입도 엔터티 역할을 하므로 자신의 속성과 관계를 가질 수 있고, 내부에 또다시 허브타입들을 가질 수 있다.

7. 물리 데이터 모델의 테이블과 데이터베이스 테이블을 비교 설명하라.

개념상 동일 하지만, 엄밀히 말하면 서로 다르다. 물리 데이터 모델에서의 테이블은 모델상에서의 추상적인 테이블인데 반해, 저장된 데이터베이스에서의 테이블은 실제 디스크에 저장되어 실체가 있는 모습의 테이블이기 때문이다. 
동일한 개념의 테이블 인스턴스가 있다. 그리고 제약조건들도 동리하게 적용된다.

8. 데이터를 저장하는 방법에 따른 테이블들을 종류별로 상세하게 설명하라
테이블에 저장하는 방식에는 분리형 테이블 방식, 일체형 테이블, 클러스터형 테이블, 파티션 테이블 방식이 있다.

분리형 테이블은 어떤한 컬럼에 의해서도 정령 되어 저장되지 않는 유형이다. 분리형 테이블은 입력되는 행들이 입력되는 순서대로 저장되기 때문에 테이블에 저장된 어떠한 컬럼도 우선순위가 없으며, 컬럼들을 액세스 할 때 모두 동등한 조건에 있다.

일체형 테이블은 인덱스 데이터를 테이블에 함께 저장하는 방식으로, 인덱스 일체형 테이블이라 부르기도 한다. 일체형 테이블은 기본키에 대하여 먼저 B* 트리 인덱스를 구성하고, 인덱스의 말단 노드에 데이터 행들을 저장하는 방식이다. 따라서 기본키를 중심으로 액세스하고자 할 때는 속도가 빠르지만, 일반 컬럼들을 액세스할 때는 분리형 테이블과같이 비정렬 저장방식의 속도와 동일하다.

클러스터형 테이블은 클러스터들에 데이터를 저장하는 방식이다. 클러스터란 컬럼 값이 동일한 행들을 저장하는 단위 저장 공안을 말한다.

파티션 테이블 저장 기법은 하나의 테이블을 분류 기준에 따라 여러 개의 파티션으로 나누고, 분류 기준에 따라 데이터들ㅇ을 자동 분류하여 해당 파티션에 저장하는 기법이다.

9. B* 트리의 비말단 노드와 말단 노드의 구조에 대하여 설명하라
비말단 노드는 데이터들과 하위의 블록 노드 포인터들을 가지고 있지만, 말단 노드는 탐색키 값과 ROWID가 한 쌍인 인덱스 엔트리들을 가지고 있다.

10. B* 트리 인덱스를 액세스 하여 테이블 데이터를 검색하는 방법에 대하여 설명하라

먼저 인덱스를 검색하여 인덱스 컬럼 값을 가진 행에 해당하는 ROWID를 찾고, 테이블에서 ROWID에 해당하는 데이터를 검색하여 결과 데이터를 반환하는 과정으로 진행된다.

11. B트리와 B+트리 및 B*트리를 비교 설명하라
B+ 트리와 B* 트리는 구조가 동일하지만, B* 트리는 초기에 각 노드들을 2/3 채우는 반면, B+ 트리는 각 노드들을 1/2까지만 채운다. B트리는 B+트리와 동일하게 초기에 각 노드들을 1/2까지 채우지만 구조가 B+트리와 조금 다르다. B트리는 비말단 노드가 데이터 블록 포인터도 함께 가지고 있기 때문에 해당 노드를 사용하여 직접 해당 데이터를 액세스할 수 있다. 그래서 B트리는 컬럼 값마다 액세스 하는 속도가 다르다. 그리고 B트리의 팬 아우트 값이 B+트리나 B+트리 보다 적기 때문에 평균적으로 속도도 느리다. 이와 같은 이유로 인해 B트리보다 B+트리나 B* 트리를 일반적으로 많이 사용한다. 오라클에서는 B*트리 방식을 채용하고 있다.

12. 데이터 독립성에 대하여 설명하라.
논리적 데이터 독립성과 물리적 데이터 독립성으로 나눌 수 있다. 논리적 데이터 독립성은 외부 스키마나 애플리케이션을 변경하지 않으면서 개념 스키마를 변경할 수 있는 성질이다.

물리적 데이터 독립성은 개념 스키마는 변경하지 않으면서 내부 스키마를 변경할 수 있는 성질이다. 즉 데이터베이스 내부의 저장 구조나 액세스 방법을 변경하더라도 개념 스키마에 해당하는 데이터모델은 영향을 받지 않아야 한다.

13. 오라클의 SGA의 의미와 구성에 대하여 설명하라
SGA는 여러 사용자들의 정보들을 공유하여 저장하는 메모리 영역으로서, 공유 풀, 데이터베이스 버퍼 캐시, 리두로그 버퍼, Java 풀, 대용량 풀 등으로 구성된다.

14. 오라클의 백그라운드 프로세스들에 대하여 설명하라.
백그라운드 프로세스는 DBWn, LGWR, SMON, PMON, CKPT 등이 필수적인 프로세스이고, 나머지 프로세스들은 선택적이다.


15. 클라이언트 사용자가 서버의 데이터베이스를 액세스하는 과정에 대하여 설명하라.

데이터베이스를 시작하면 가장 먼저 인스턴스가 시작되는데, 인스턴스가 시작될 때 SGA가 할당되고 백그라운드 프로세스들이 수행된다. 사용자가 오라클 서버에 연결을 요청하면, 클라이언트의 사용자 프로세스가 시작됨과 동시에 사용자 프로세스에 PGA가 할당되고, 사용자가 세션을 설정하게 되면 서버 프로세스가 시작된다. 세션은 사용자가 오라클 서버에 연결하는 단위이며, 하나의 클라이언트에서 여러 세션을 설정할 수 있다. 물론 사용자가 연결을 종료하게 되면 PGA가 해제되고 사용자 프로세스도 종료되며, 전용 서버 방식인 경우는 사용자 프로세스가 종료되면 서버 프로세스도 함께 종료되고, 공용 서버 방식인 경우는 여러 명이 사용하는 방식이므로 서버 프로세스가 종료되지 않는다.

16. 오라클의 저장구조에 대하여 설명하고, 저장 구조의 각 요소에 대하여 설명하라.
데이터베이스, 테이블 스페이스, 데이터파일, 세그먼트, 익스텐트, 블록, os 블록이 있다.
오라클은 데이터베이스를 테이블스페이스, 세그먼트, 익스텐트 및 블록 등의 요소들을 사용하여 논리적으로 관리하면서 데이터 파일에 물리적으로 저장한다.

데이터베이스는 논리적으로 여러 개의 테이블 스페이스들이 모여 구성된다. 그리고 하나의 테이블스페이스는 논리적으로 여러 세그먼트들로 구성되지만 물리적으로 여러 데이터 파일들에 저장된다.

각 세그먼트는 논리적으로 여러 엑스텐트들이 모여서 구성되고, 익스텐트 단위로 초기화되고 증감된다.

데이터베이스는 효율적으로 데이터들을 액세스하기 위해 여러 블록들을 하나의 단위로 묶어 익스텐트 저장 공간을 확보하게 되고, 물리적인 데이터 파일의 구성 단위가 되며, 여러 개의 os블록들로 구성된다.

오라클은 디스크에 실제 데이터를 항상 블록 단위로 액세스한다.

17. 3-층 클라이언트/서버 아키텍처에 대하여 설명하라
클라이언트와 데이터베이스 서버 사이에 중간 단계를 하나 더 추가한 형태인데, 이러한 중간 단계를 응용 서버 또는 웹서버라 부른다.
클라이언트에서는 GUI와 일부 비즈니스 규칙을 가진 애플리케이션이 수행되고, 응용 서버는 데이터베이스 서버에 저장된 데이터들을 액세스하는데 필요한 비즈니스 규칙들을 저장하여 클라이언트로부터 요청받은 작업을 처리하며, 데이터베이스 명령들을 데이터베이스 서버에 전송한다. 데이터베이스 서버는 해당 작업을 DBMS를 통하여 데이터베이스에 접근하여 처리하고, 처리된 결과를 응용 서버를 통하여 클라이언트에게 전송한다.

18. SQL 문이 수행되고 처리되는 과정에 대하여 설명하라.
처음에 SQL문으로 DBMS에게 명령을 내린다. 이 때 툴을 통하여 직접 SQL문을 수행시킬 수도 있고, 애플리케이션 프로그램 내부에 SQL문을 포함시켜 임베디드 SQL 형태로 수행시킬 수도 있다. 그러면 DBMS는 해당 명령을 해것하여 실행 계획을 수립하고, 수립된 실행 계획을 실행하여 데이터베이스에 접근하고, 결과를 구하여 반환한다. 애플리케이션은 반환 결과를 확인하거나, 반환 결과를 사용하여 다른 동작을 수행한다. 사용자나 데이터베이스 관리자는 실행결과를 확인하고, 다른 조치를 취할 수 있다.
